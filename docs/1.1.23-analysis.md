### 📌 模块 3：业务核心 —— 决策弹窗 Bug 排查（**关键修订**）

#### 3.1 用户报告的 Bug
- **Bug 1**：cc 弹决策面板时，飞书**收不到**决策卡片
- **Bug 2**：用户即便在飞书侧回复"1"/"yes"，也**无法**把决策回传到 cc 进程

#### 3.2 完整链路诊断

```
✅ cc 弹面板（TUI 显示 ╭──╮ 框）
✅ extractChoicePanel 正则识别（cc-adapter.ts:99-175）
✅ session.awaiting 状态标记（state.ts:159-162）
✅ onAwaiting 回调触发（state.ts:161）
✅ buildAwaitingCard 构建卡片（sender.ts:607-639）
✅ sendCard 调用飞书 API
❌ 卡片到达飞书但「无按钮」→ 用户看到的只是一段纯文本
❌ 即使用户手打回复，ws-client 未注册卡片回调事件
❌ handleMessage 不区分"普通提问"和"决策回复"
❌ sendChoice 在 cc-adapter.ts:189 定义，全仓只有一处理论调用但实际未被触发
```

#### 3.3 两个 Bug 的精确根因

##### **Bug 1：卡片"长得像"但飞书认不出是交互卡**

**位置**：`src/server/channel/feishu/sender.ts` `buildAwaitingCard`（607-639 行）

**根因**：
- `InteractiveCard` 的 elements 类型限定为 `MarkdownElement | HrElement | NoteElement | TableElement`（sender.ts:106）
- **完全没有 button / action 元素**
- 飞书卡片要触发回调，**必须**带 `button` element 且配 `action` 字段
- 当前实现就是把决策内容塞进 markdown 里，让飞书显示一段"请回复 1/2/是/否"的文本提示

**后果**：
- 用户看到的就是普通消息卡片，没有"点 1"或"点 2"的按钮
- 即使用户认真看到提示文字、手打回复，下一步也走不通（见 Bug 2）

##### **Bug 2：飞书回复路径完全没接上**

**位置**：`src/server/channel/feishu/ws-client.ts` `KNOWN_EVENTS`（101-112 行）+ `handleMessage`（320-422 行）

**根因 A**：飞书 WebSocket 监听的事件清单里**没有** `im.message.card.clicked_v1`。即使 Bug 1 修了把按钮加上、用户点了按钮、飞书发了卡片回调事件，本服务也不接。

**根因 B**：用户用"普通文本消息"回复"1"或"yes"时，消息会进 `handleMessage`，但该函数**只有一条路径**：把消息当新输入 `adapter.sendInput(...)` 发给 cc tmux。**没有在前面加判断**：
```ts
const s = getSession(processName);
if (s && s.awaiting) {
  // 这是用户对决策面板的回复 → 走 sendChoice
  return adapter.sendChoice(sessionName, msgText, s.awaiting.panel, executor);
}
// 否则才走 sendInput
```

**根因 C**：`sendChoice` 已经在 `cc-adapter.ts:189-219` 完整实现（支持 "1"/"yes"/"是"/关键词匹配），但**全仓 grep 找不到任何调用方**，是一个孤儿函数。

#### 3.4 修复方案（双轨并行，保证可用性）

考虑到飞书卡片按钮回调有飞书侧配置成本（机器人需开通 `im:resource` 权限 + 卡片回调订阅），**两条路径同时做，文本回退打底**：

##### **路径 A（必做，简单可靠）：文本消息识别**
在 `ws-client.ts handleMessage` 入口加判断：
```
收到消息 → 查 getSession(processName)
  ├─ session.awaiting != null
  │  └─ 调 adapter.sendChoice(sessionName, msgText, panel, executor)
  │     成功 → 发"已选择 N. xxx"回执 + 清掉 session.awaiting
  │     失败 → 发"无法识别您的选择，请回复 1/2 或 yes/no"
  └─ session.awaiting == null
     └─ 走原 sendInput 流程
```
改动量：ws-client.ts 加 20 行；其余不动。

**用户体验**：在飞书直接打"1"或"yes"或"是"，cc 那边自动收到对应按键 + 回车，决策继续。无需点按钮、不依赖飞书后台配置。

##### **路径 B（可选，体验更好）：补按钮 + 接卡片回调**
1. `sender.ts buildAwaitingCard` 加 `action` 元素，把每个选项做成按钮（飞书 message card 的 button 元素，`value: {action:'cc_choice', processName, index}`）
2. `ws-client.ts KNOWN_EVENTS` 加 `im.message.card.clicked_v1`
3. handleMessage 同级加 handleCardAction：从 `value.processName + value.index` 找回 session.awaiting.panel，调 sendChoice

改动量：sender.ts 加 30 行（按钮 element 类型扩展 + 重写 buildAwaitingCard）；ws-client.ts 加 30 行（事件分发 + handleCardAction）；类型定义 5 行。

##### **顺带要修的副作用**
- `state.ts:148` "已经回复并结束 → 停轮询" 的判断目前是 `session.replied && !session.awaiting`，意味着面板出现期间 polling 一直跑——OK，符合设计
- 但 `startProgressTimer`（state.ts:82）的 60s 进度卡片当 `session.awaiting` 为真时**不应**继续发"处理中"卡片，避免和决策卡片混叠（ws-client.ts:359 已经判断过 `if (s.awaiting) return`，这里已 OK）
- **`hardDeadlineTimer`（10 分钟硬超时）在 awaiting 状态下应豁免**：ws-client.ts:367-387 已经判断 `if (s.awaiting) { ... 跳过硬超时 }` ✅
