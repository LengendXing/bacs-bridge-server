# v1.1.3 · Web Terminal 技术方案

> 分支：`feat/v1.1.3-web-terminal`
> 起草：2026-05-12

## 目标

在「绑定」列表的「操作」一列，新增 **「Web Terminal」** 按钮（与 Attach 并列）。点击后**在浏览器内打开一个 xterm 终端，直接进入对应绑定进程的 tmux 会话**（cc / codex 当前正在跑的那个 pane），用户可以像本地 `tmux attach -t cc-xxx` 一样交互。

**关键约束（核心难点）：关闭 web-terminal 窗口 ≠ kill 业务进程。**
原 cc/codex 进程必须继续跑，飞书侧消息桥不受影响。

---

## 现状梳理

### 当前架构相关点
- 业务进程：`tmux new-session -d -s cc-{processName} "bash -ilc '...; exec claude ...'"`，**detached** 模式启动
- 绑定记录：`bindings.processName` + `bindings.machineId`（builtin = 本机，否则远程 SSH）
- Executor 抽象（`src/server/executor/types.ts`）：
  - `LocalExecutor` 走 `child_process.exec`
  - `SshExecutor` 走 ssh2，已有 `Client.exec()` 用于一次性命令
  - 已有方法：`exec / capturePane / sendInput / sendKeys / killSession`
- 当前 Attach 按钮：只是 `navigator.clipboard.writeText('tmux attach -t cc-xxx')` 让用户去本地开 SSH

### 飞书桥的轮询不会和 web-terminal 冲突
飞书侧用 `tmux capture-pane -p` 抓输出 + `tmux send-keys` 写输入；和"另一个客户端 attach 到同一个 tmux session"是 tmux 原生支持的多客户端共享 pane —— 不会互踢。这是用 tmux 的最大好处之一，方案可以直接利用。

---

## 关键设计决策

### 1. 用 tmux attach 而不是 tmux switch / new-window

- ✅ `tmux attach -t cc-xxx`：把当前 PTY 接入已有 session，**detach（按 Ctrl-b d 或断开 PTY）只断客户端，不杀 session**
- ❌ `tmux kill-session`：直接干掉业务进程，违反约束
- ❌ 起一个新 shell 然后 `cd` 过去：根本不是同一个 tmux session

### 2. detach 行为如何保证「关窗口 ≠ kill 进程」

tmux 的 detach 语义已经天然满足要求，三层保险：

1. **PTY 进程退出方式**：客户端断开（关 WebSocket）后，服务端 PTY 子进程（`tmux attach`）收到 SIGHUP，tmux 客户端进程退出，**业务 session 保留**（这是 tmux detached session 的核心特性）
2. **绝对禁止 kill-session**：web-terminal 模块**只读地**使用 `tmux attach`，不写 `tmux kill-session`；卸载/解绑功能在别的路径上
3. **代码层防御**：终端 spawn 时显式 `bash -c "exec tmux attach -t {session}; exit"`，不嵌任何 kill 逻辑；任何"关闭终端"路径只 close PTY，不 send `tmux kill-*`

### 3. 本机 vs 远程：两条 PTY 通道

| 机器 | 服务端实现 | 备注 |
|---|---|---|
| 本机（builtin） | `node-pty` spawn `bash -c "exec tmux attach -t cc-xxx"` | 需新增依赖 `node-pty` |
| 远程（SSH） | 复用 `SshExecutor` 的 ssh2 `Client.shell()` 开远程 PTY，进 shell 后立刻 `tmux attach -t cc-xxx` | ssh2 已经支持 PTY，零新依赖 |

**为什么本机也要 PTY 而不是简单 `child_process.spawn`：**
xterm 终端必须有真正的 TTY 才能正确处理光标定位、ANSI 转义、SIGWINCH（窗口 resize）。`child_process.spawn` 拿到的是普通 pipe，cc 这种 TUI 会渲染成乱码。`node-pty` 是绕不过去的依赖。

### 4. WebSocket 协议

新增端点：`GET /ws/terminal?bindingId=123&token=<JWT>`

**协议：二进制透传 + 控制帧**

| 客户端 → 服务端 | 服务端 → 客户端 |
|---|---|
| 二进制：键盘字节流（写入 PTY stdin） | 二进制：PTY stdout/stderr 字节流（写入 xterm） |
| 文本 JSON `{type:"resize", cols, rows}`：窗口大小同步 | 文本 JSON `{type:"error", message}`：错误提示 |
|  | 文本 JSON `{type:"ready"}`：PTY 启动成功 |

**鉴权**：`?token=<JWT>` query 参数（沿用 v1.0.5 SSE 同款方案，复用 `verifyToken`）。WebSocket 不能加自定义 header，必须走 query。

### 5. 安全边界

- ⚠️ **Web Terminal 拥有的权限 = 跑桥接服务的进程权限**。本机模式下相当于把服务器 root（如果服务用 root 跑）暴露给浏览器；远程模式下相当于把绑定机的 SSH 账号暴露给浏览器
- 必须 `requireAuth`：只允许已登录的管理员
- 写一条**审计日志**：开启 / 关闭都记 `audit_logs`（who / when / which binding / IP）
- 限并发：每个 binding 最多 N 个活跃终端会话（建议 N=2），防止滥用占资源
- 命令注入防御：`bindingId` 走 SQL 参数化查询，`processName` 在拼 `tmux attach -t {name}` 前用白名单正则校验（已知项目里 processName 由后端自己生成，但仍要防御）

### 6. 多客户端共享同一个 tmux session 的副作用

tmux 默认行为：多个客户端 attach 同一 session 时，**所有人的窗口大小取最小值**。这意味着：
- 飞书桥用 `capture-pane` 不开 PTY，不影响 size
- 但如果两个 web-terminal 同时 attach，pane 会被拉到两人窗口的最小值

解决方案：每个终端连接开 `tmux new-session -t cc-xxx -s cc-xxx-view-{随机}` 创建一个**新 session 共享同一组 windows**（tmux 的 grouped session 特性），各客户端独立 size。代价：`-s` 新建的 view session 需要在 PTY 退出时清理，否则 `tmux ls` 会留垃圾。

简化版（推荐 v1.1.3 先做）：直接 `tmux attach -t cc-xxx`，接受多客户端 size 共享的代价；同一 binding 限制 1 个活跃终端。如果实际有多人同时操作的需求，下一版再上 grouped session。

## 实现拆分

### 后端

**新增文件 `src/server/terminal/pty-bridge.ts`**

抽象一个 `TerminalSession` 接口，本机 / 远程两套实现：

```ts
interface TerminalSession {
  write(data: Buffer | string): void;     // 客户端 → PTY
  resize(cols: number, rows: number): void;
  onData(cb: (chunk: Buffer) => void): void;  // PTY → 客户端
  onExit(cb: (code: number | null) => void): void;
  close(): void;  // 关闭 PTY，不影响业务 session
}

function openLocalTerminal(sessionName: string, cols: number, rows: number): TerminalSession;
function openSshTerminal(executor: SshExecutor, sessionName: string, cols: number, rows: number): TerminalSession;
```

**本机实现要点**
- `import * as pty from 'node-pty'`
- `pty.spawn('bash', ['-ilc', `exec tmux attach -t ${quote(sessionName)}`], { name: 'xterm-256color', cols, rows, cwd: process.env.HOME, env: process.env })`
- `bash -ilc` 加 `exec` 替换进程，关 PTY 时直接走 SIGHUP，tmux 干净 detach（与 cc-adapter 启动套路一致）

**远程实现要点**
- 复用 `SshExecutor` 的 ssh2 `Client`，但**单独 `client.shell({ term: 'xterm-256color', cols, rows })` 拿一个独立 channel**（不影响现有 exec 池）
- shell 通道开起来后第一条命令：`exec tmux attach -t {name}\n`（`exec` 同样让 bash 被 tmux 替换，PTY 一关就 detach）
- 注意：当前 SshExecutor 的 `client` 实例是连接复用的，开 shell channel 不会冲突；如果担心阻塞 exec，下一阶段可以为 terminal 单开一个 ssh2 Client

**新增文件 `src/server/routes/terminal.ts`**

挂在 Express 上但用作 WebSocket upgrade 端点。其实更干净的做法是直接在 `src/server/index.ts` 里把 ws server 挂到 `/ws/terminal` 路径。复用现有的 `WebSocketServer`（如已有），或者新建一个 `WebSocketServer({ noServer: true })` 然后在 `server.on('upgrade')` 里按路径分发。

流程：
1. `upgrade` 事件取 `?bindingId=&token=`
2. 校 token → 校 binding 存在 → 解析 machineId → 拿 executor
3. `tmux has-session -t cc-{name}` 预检（避免 attach 到不存在的 session）
4. 开 PTY/shell channel，两端 pipe（PTY → ws.send；ws.message → pty.write）
5. ws close / pty exit / 异常任一发生 → 关另一端，**不触发 tmux kill**
6. 写审计日志（开始 + 结束 + 持续时长）

### 前端

**新增文件 `src/client/views/TerminalView.vue`**（独立路由，不嵌在 BindingsView 里）

- 新路由：`/terminal/:bindingId`
- 引入 `xterm` + `xterm-addon-fit` + `xterm-addon-web-links`
- 全屏布局，顶部一行简洁状态条（绑定名 / 机器 / 连接状态 / 关闭按钮）
- 关闭按钮 = `window.close()`（如果是新窗口打开的）或 `router.back()`
- WebSocket 用 `new WebSocket('/ws/terminal?bindingId=X&token=Y')`，加 `binaryType = 'arraybuffer'`
- onResize（fit addon）→ 发 `{type:'resize', cols, rows}`
- 重连策略：网络断开时显示 banner "已断开，业务进程仍在运行"，提供「重新连接」按钮

**修改 `src/client/views/BindingsView.vue`**

操作列加一个按钮：
```vue
<button class="btn-mac btn-mac-sm" @click="openTerminal(b)" title="在浏览器内打开 tmux 终端">Terminal</button>
```

`openTerminal` 用 `window.open('/terminal/' + b.id, '_blank', 'width=1200,height=800')` —— 新窗口避免误关 SPA、不浪费主页面布局，关窗口语义清晰。

## 失败模式 & 防御

| 场景 | 表现 | 处置 |
|---|---|---|
| `tmux has-session` 失败（业务 session 已死） | attach 会立刻 exit | ws 发 `{type:'error',message:'业务进程未运行，请先 mount/重启绑定'}` 后关 ws |
| node-pty 在生产机未预编译 | 服务启动报错 | 在 `package.json` 加 `optionalDependencies`，或加运行时检测——若 node-pty 加载失败，本机 terminal 按钮 disable + tooltip 提示 |
| SSH 通道断 | xterm 卡住，看不到提示 | ssh2 `client.on('close')` → 关 ws；前端 onclose 显示重连 banner |
| 用户在 web-terminal 里手贱执行 `tmux kill-session -t cc-xxx` | 业务进程真的被 kill | **这是用户主动行为，不防**。但日志里能查到。文档提示风险即可 |
| 用户按 `Ctrl-b d` 主动 detach | tmux 客户端退出 → PTY 退出 → ws 关 | 这就是预期行为，前端显示「已退出（业务进程仍在运行）」 |
| WS 消息洪水 | 服务端被打爆 | xterm 输入侧 throttle / 服务端 ws 加 `maxPayload` 限制 |

## 测试计划

1. **本机 happy path**：建一个 cc 绑定，开 web-terminal，看到 cc TUI；输入消息得到回复
2. **关窗口不杀进程（核心）**：开 terminal → 在 cc 里发一句话回车 → 关 webterminal 窗口 → 飞书侧发消息，验证 cc 仍能响应
3. **远程 happy path**：同上但 binding 是远程机
4. **同时多个客户端**：终端打开期间，飞书发消息照常工作（飞书 capturePane 应能看到 web-terminal 里输入的内容）
5. **业务 session 不存在**：手动 `tmux kill-session -t cc-xxx` 后再开 terminal，应得到 error 提示而不是卡死
6. **审计日志**：每次开/关都进 audit_logs

单元测试覆盖：
- `pty-bridge.openLocalTerminal` 启动命令拼接正确（`bash -ilc 'exec tmux attach -t {名字}'`）
- session 名注入防御（特殊字符 reject）
- 关闭路径不调用 `kill-session`（grep 断言或 mock executor）

## 待与需求方确认

1. **多终端策略**：v1.1.3 是否接受「每个 binding 同一时刻只允许 1 个 web-terminal」简化版？多人协作可下个版本上 grouped session
2. **打开方式**：新窗口 `window.open` 还是 SPA 内嵌路由？我推荐**新窗口**（关窗口语义直观，不影响 SPA 状态）
3. **是否需要「只读模式」**：有些团队希望日志查看不要赋写权限。v1.1.3 默认全交互，只读模式留作未来开关
4. **node-pty 部署成本**：项目当前是否能接受新增编译型依赖？若部署机环境受限，可降级为「仅远程机器支持 web-terminal，本机不支持」（远程走 ssh2 不需要 node-pty）
